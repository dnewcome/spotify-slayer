# Devlog Б─■ 2026-03-29

## What we built

### The app

Started from scratch: a Next.js 16 web app that connects to Spotify and lets you build DJ sets. The goal is a personal tool that solves real problems in a DJ prep workflow Б─■ discovering tracks, capturing them without losing them to the wrong playlist, tagging them with energy and feel, and eventually bridging the gap to file acquisition via Soulseek.

### Auth & Spotify integration

Got Spotify OAuth working end-to-end with next-auth. This took longer than expected Б─■ Spotify deprecated `localhost` redirect URIs in early 2026 (now rejects them as "Insecure"), requiring `http://127.0.0.1` instead. Also hit a breaking API change: `/v1/playlists/{id}/tracks` was deprecated in February 2026 in favor of `/v1/playlists/{id}/items`, which also changed the track key from `track` to `item` and moved `tracks.total` to `items.total` on simplified playlist objects. Got all of that sorted.

Dev environment quirk: SWC's native arm64 binary was corrupted, so the dev server runs with `--webpack` flag via `node node_modules/next/dist/bin/next dev --webpack`. Also needed `allowedDevOrigins: ["127.0.0.1"]` in `next.config.ts` for HMR to work Б─■ without it, JavaScript hydration silently fails and buttons are unclickable.

### Core features

- **Playlists** Б─■ browse all your Spotify playlists, see track count and artwork
- **Playlist detail** Б─■ full track listing with "Add to Set" on every track
- **Recently played** Б─■ home page shows last 20 tracks with timestamps, all addable to sets
- **Search** Б─■ full Spotify track search, add results directly to sets
- **DJ Sets** Б─■ create, rename, delete sets; drag-and-drop reorder tracks; total duration display
- **Nav** Б─■ profile photo, username, sign out

### Track library & metadata

Introduced a separate `track_library` localStorage store keyed by Spotify track ID. Metadata persists across sets Б─■ tag a track once and it carries those tags everywhere.

Per-track fields: `energyLevel` (1Б─⌠5), `busyness` (1Б─⌠5), `tags`, `genre`, `isrc`, `mbid`, `mbEnriched`, `acquisitionStatus`, `localPath`, `notes`.

Energy scale: 1 = ambient/sparse, 5 = peak/banger.
Busyness scale: 1 = minimal space, 5 = dense/complex.

### MusicBrainz enrichment

When a track is added to a set, the app fires a background call to `/api/enrich`. The route:
1. Fetches the full Spotify track server-side to get `external_ids.isrc` (recently played uses simplified track objects that omit this)
2. Looks up the ISRC on MusicBrainz to get the recording MBID
3. Fetches recording genres + tags
4. Fetches the primary artist's genres + tags (much better populated than recording-level)
5. Merges and deduplicates, stores in localStorage

When you open a set, any tracks that haven't been enriched yet are processed automatically in the background, one per second to respect MusicBrainz rate limits. Genre pills appear on track rows as enrichment completes. Tracks not found in MusicBrainz are silently skipped.

Discovered along the way: MusicBrainz ISRC endpoint returns 404 (not empty array) for unknown ISRCs, and doesn't accept `inc=recordings` as a query parameter (recordings are returned by default).

### Energy arc visualization

The set builder now has two visual layers:

**Per-row color stripe** Б─■ a 4px left border on each track row encodes energy and busyness as a color using HSL: hue maps energy from blue (1) through green to red (5), saturation/lightness maps busyness from dim to bright. Untagged tracks show as zinc gray.

**Arc bar** Б─■ a horizontal color strip at the top of the set builder. Each track is a block proportional to its duration. Shows the energy shape of the whole set at a glance. A set with a good arc should run blue Б├▓ green Б├▓ yellow Б├▓ red and come back down.

**Selection** Б─■ clicking a track row highlights the corresponding block in the arc bar with a white outline (rendered as an absolutely positioned sibling outside the `overflow-hidden` container, which was clipping outline-based approaches). Click again to deselect.

### Data model decisions

- `DJSetTrack` has a `slotId` (UUID) separate from `id` (Spotify track ID). This allows the same track to appear multiple times in a set without React key collisions or dnd-kit conflicts. Existing sets in localStorage are migrated automatically on first load.
- MusicBrainz enrichment is fire-and-forget, non-blocking. UI doesn't wait for it.
- Enrichment won't retry a track that came back `not_found` Б─■ `mbEnriched: true` is set regardless of outcome to prevent hammering the API.
